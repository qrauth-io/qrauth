QRVA Protocol ​

QRVA (QR Verification & Attestation) is the cryptographic protocol underlying every QRAuth-signed QR code. It provides end-to-end integrity: from the moment a QR code is issued to the moment a scanner verifies it, no party can alter the code's content without invalidating the signature.

Signature scheme ​

Algorithm: ECDSA with P-256 (prime256v1, secp256r1), SHA-256 digest

Key format: PEM — PKCS#8 private key, SPKI public key

Key lifecycle:

  • One signing key pair per organization
  • Key ID is a UUID v4; referenced by kid in the signature metadata
  • Key rotation: generate a new pair, update the active key pointer — old signatures remain verifiable against the archived public key

Canonical payload ​

The payload that is signed is deterministic — identical inputs always produce identical bytes. This matters because PostgreSQL's JSONB column type reorders keys alphabetically, so naive JSON.stringify would produce different bytes depending on insertion order.

ts
// From packages/shared/src/crypto.ts
export function hashPayload(
  token:       string,   // Short token embedded in QR image
  url:         string,   // Destination URL
  geoHash:     string,   // Geohash of registered location, or '' if none
  expiry:      string,   // ISO 8601 expiry datetime, or '' if no expiry
  contentHash: string,   // SHA-256 of stableStringify(content), or ''
): string {
  return `${token}:${url}:${geoHash}:${expiry}:${contentHash}`;
}

Fields are colon-delimited. Empty optional fields contribute an empty segment, preserving positional unambiguity. The resulting string is hashed with SHA-256 before signing.

Content hash for rich content types ​

For non-URL QR codes (vCard, Event, Coupon, etc.) the content field is a JSON object. To produce a deterministic hash across different JSON serialisers:

ts
// From packages/shared/src/crypto.ts
export function stableStringify(value: unknown): string {
  if (Array.isArray(value)) {
    return '[' + value.map(stableStringify).join(',') + ']';
  }
  if (value !== null && typeof value === 'object') {
    const obj = value as Record<string, unknown>;
    const keys = Object.keys(obj).sort();
    return '{' + keys.map(k => JSON.stringify(k) + ':' + stableStringify(obj[k])).join(',') + '}';
  }
  return JSON.stringify(value);
}

// Usage
const contentHash = sha256(stableStringify(content)); // hex string
const payload     = hashPayload(token, url, geoHash, expiry, contentHash);

WARNING

Always use stableStringify when computing or verifying content hashes. Standard JSON.stringify will produce wrong results on objects whose keys are not already sorted.

Full signing flow (server) ​

ts
// 1. Compute canonical payload
const contentHash = content ? sha256(stableStringify(content)) : '';
const payload     = hashPayload(token, destinationUrl, geoHash, expiry, contentHash);

// 2. Hash the payload
const payloadHash = sha256(payload); // hex

// 3. Sign with ECDSA-P256
const signer = crypto.createSign('SHA256');
signer.update(payloadHash, 'hex');
signer.end();
const signature = signer.sign(privateKeyPem, 'base64url');

// 4. Store { token, signature, contentHash, ... } in DB

Verification flow ​

On each scan at https://qrauth.io/v/:token:

1. Signature check
   Reconstruct canonical payload → re-hash → verify ECDSA signature
   against org's active public key.
   ↓ fails → INVALID_SIGNATURE → trust score = 0

2. Expiry check
   If qrCode.expiresAt is set and is in the past → EXPIRED

3. Transparency log check
   Confirm the signature hash appears in the append-only log
   at the expected index.
   ↓ missing → TRANSPARENCY_LOG_MISMATCH → trust score −20

4. Fraud analysis
   Run 6 inline signals + dynamic rules → compute trust score

5. Render verification page
   trustScore ≥ 80 → green reveal
   50–79 → amber caution
   < 50  → red failure
ts
import { QRAuth } from '@qrauth/node';

const qrauth = new QRAuth({ apiKey: process.env.QRAUTH_API_KEY! });

const result = await qrauth.verify('AbCdEfGh', {
  clientLat: 37.9838,
  clientLng: 23.7275,
});

result.verified;                              // boolean
result.security.signatureValid;               // ECDSA check
result.security.transparencyLogVerified;      // log inclusion check
result.security.trustScore;                   // 0–100

Transparency log ​

Every QR code issuance is appended to an organization-scoped, append-only transparency log. The log entry contains:

FieldValue
indexMonotonically increasing integer
tokenQR token
payloadHashSHA-256 of the canonical payload
signatureECDSA signature (base64url)
createdAtISO 8601 timestamp

The transparency_log_index is returned when a QR code is created and is embedded in the QR code URL for independent auditability.

Public transparency endpoint ​

GET /api/v1/transparency/:orgSlug/:index

Returns the log entry at that index. Anyone can retrieve and independently verify any signature in the log — no authentication required.

ts
const entry = await fetch(
  'https://qrauth.io/api/v1/transparency/my-org/42'
).then(r => r.json());

// entry.token, entry.payloadHash, entry.signature, entry.createdAt

Content types and signing ​

Each content type has a Zod schema defined in packages/shared/src/content-types/. The schema validates the content object before signing. If the content fails validation, the QR code is rejected.

Content typePayload fields
urldestinationUrl (no content object)
vcardname, phone, email, org, url
eventtitle, start, end, location, url
coupontitle, discount, code, expiry
pdfassetId (uploaded to content asset store)
feedbackformId, redirectUrl

Verification via the browser SDK ​

The verification page JavaScript independently re-validates the origin before rendering:

js
// Origin integrity check (runs in the browser on the verification page)
const expected = 'qrauth.io';
if (window.location.hostname !== expected) {
  // Render phishing warning — page is being served from the wrong domain
  document.body.innerHTML = PHISHING_WARNING_HTML;
  throw new Error('Origin mismatch');
}

This prevents a cloned verification page from rendering normally even if all assets were faithfully copied, because window.location.hostname will be the phishing domain.
